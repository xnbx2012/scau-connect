## 二、CAS 认证流程

### 2.1 认证序列

aTrust 的 CAS 认证流程由 7 个步骤组成：

```
步骤 1: GET /passport/v1/public/authConfig
        ↓ 获取 csrf_token 和初始 sid cookie

步骤 2: Selenium 无头浏览器访问 /passport/v1/public/casLogin
        ↓ 填写用户名密码，浏览器自动处理 RSA 加密

步骤 3: 浏览器跳转到 portal/shortcut.html
        ↓ 触发 JS 链：authConfig?mod=1 → authCheck → ticketExchange

步骤 4: GET /passport/v1/public/authConfig?mod=1
        ↓ 注册 CAS 登录，获取新的 csrf_token

步骤 5: GET /passport/v1/auth/authCheck
        ↓ 获取 sidTicket

步骤 6: POST /passport/v1/public/ticketExchange
        ↓ 交换 ticket 完成会话建立

步骤 7: GET /passport/v1/user/onlineInfo
        ↓ 确认 isOnline = true

步骤 8: POST /controller/v1/user/clientResource
        ↓ 获取资源信息（DNS、路由、IP 段等）
```

### 2.2 关键 API 端点

所有 API 请求都需要携带以下查询参数：
```
clientType=SDPBrowserClient&platform=Windows&lang=zh-CN
```

| API | 方法 | 说明 |
|-----|------|------|
| `/passport/v1/public/authConfig` | GET | 获取 csrf_token 和初始 sid cookie |
| `/passport/v1/public/casLogin` | GET | CAS 登录页面 URL |
| `/passport/v1/auth/cas?ticket=XXX` | GET | CAS ticket 回调 |
| `/passport/v1/auth/authCheck` | GET | 获取 sidTicket |
| `/passport/v1/public/ticketExchange` | POST | 交换 sidTicket 完成会话 |
| `/passport/v1/user/onlineInfo` | GET | 查询会话状态，isOnline=true 表示在线 |
| `/controller/v1/user/clientResource` | POST | 获取资源信息（IP 段、节点地址等） |

### 2.3 Selenium 认证实现

#### 为什么使用 Selenium？

aTrust 的密码字段使用 **RSA 加密**：密码在浏览器端通过 JavaScript 使用公钥加密后传输。要正确登录，需要：

1. 获取 RSA 公钥（从登录页面 JS 中提取）
2. 实现 RSA 加密逻辑（需要 rsa 或 cryptography 库）
3. 构造正确的加密格式

使用 Selenium 可以**完全绕过**密码加密：浏览器自己处理 RSA 加密，我们只需要填写明文密码即可。

#### CDP 获取 HttpOnly Cookie

Selenium 标准 API 无法读取 HttpOnly cookie（如 CASTGC、sid），但通过 Chrome DevTools Protocol (CDP) 可以获取：

```python
# 启用 CDP
driver.execute_cdp_cmd("Network.enable", {})

# 获取所有 cookie（包括 HttpOnly）
result = driver.execute_cdp_cmd("Network.getAllCookies", {})
raw_cookies = result.get("cookies", [])
```

CDP 返回的 cookie 名称格式为 `CASTGC_-_vpn.scau.edu.cn`，需要去除域名后缀得到标准名称 `CASTGC`。

#### Portal Shortcut 页面

CAS 登录成功后，需要访问 `/portal/shortcut.html`。这个页面的 JavaScript 会自动执行：
1. `authConfig?mod=1` - 注册 CAS 会话
2. `authCheck` - 获取新的 sid
3. `ticketExchange` - 完成会话建立

完成后会设置 `online=1` cookie，表示用户已在线。

### 2.4 会话状态

认证成功后，Session 对象包含以下关键数据：

```python
session.cookies = {
    "sid": "...",           # 会话 ID，节点验证用
    "sid.sig": "...",       # sid 签名
    "CASTGC": "...",        # CAS TGC，保留 CAS 会话
    "online": "1",          # 在线状态
}
session.csrf_token = "..."  # CSRF token
session.is_online = True    # 是否在线
```

## 三、HTTP 代理实现

### 3.1 本地代理架构

```
客户端                              本地代理 (127.0.0.1:1081)
  │                                         │
  │─── HTTP 请求 ─────────────────────────▶│
  │                                         │
  │                                         ▼
  │                              ┌──────────────────────┐
  │                              │  解析请求，提取      │
  │                              │  Host 和路径         │
  │                              └──────────────────────┘
  │                                         │
  │                                         ▼
  │                              ┌──────────────────────┐
  │                              │  重写 Host + Cookie   │
  │                              │  发送到 aTrust        │
  │                              │  Web Proxy           │
  │                              └──────────────────────┘
  │                                         │
  │                                         ▼
  │                              aTrust Web Proxy
  │                              (*.s.vpn.scau.edu.cn)
  │                                         │
  ◀─────────────────────────────────────────
```

### 3.2 HTTPS CONNECT + MITM

对于 HTTPS 请求，代理使用 **HTTP CONNECT + MITM** 方式：

1. **客户端发送**：`CONNECT www.example.com:443 HTTP/1.1`
2. **代理返回**：`HTTP/1.1 200 Connection Established`
3. **代理返回伪造证书**：使用本地 CA 为目标域名签发证书
4. **客户端与代理建立 TLS**：看到的是伪造的证书
5. **代理与目标服务器建立 TLS**：使用真实请求
6. **代理在中间解密/重加密**：转发请求和响应

### 3.3 本地 CA 证书

本地 CA 证书保存在 `.proxy-ca/local-ca.crt.pem`，每个目标域名的 leaf 证书保存在 `.proxy-ca/leaf/<sha256>.crt.pem`。

证书生成使用 Python 的 `cryptography` 库：

```python
from cryptography import x509
from cryptography.hazmat.primitives import hashes, serialization
from cryptography.hazmat.primitives.asymmetric import rsa
from cryptography.x509.oid import NameOID

# 1. 生成 CA 私钥
ca_key = rsa.generate_private_key(public_exponent=65537, key_size=2048)

# 2. 生成 CA 证书（自签名）
ca_cert = (
    x509.CertificateBuilder()
    .subject_name(x509.Name([x509.NameAttribute(NameOID.COMMON_NAME, "scau-connect MITM CA")]))
    .issuer_name(x509.Name([...]))  # 自签所以 issuer = subject
    .public_key(ca_key.public_key())
    .serial_number(x509.random_serial_number())
    .not_valid_before(datetime.utcnow())
    .not_valid_after(datetime.utcnow() + timedelta(days=365))
    .sign(ca_key, hashes.SHA256())
)

# 3. 为目标域名签发 leaf 证书
leaf_key = rsa.generate_private_key(...)
leaf_cert = (
    x509.CertificateBuilder()
    .subject_name(x509.Name([x509.NameAttribute(NameOID.COMMON_NAME, target_domain)]))
    .issuer_name(ca_cert.subject)  # 用 CA 签名
    .public_key(leaf_key.public_key())
    ...
    .sign(ca_key, hashes.SHA256())
)
```

### 3.4 请求转发到 aTrust Web Proxy

aTrust Web Proxy 根据 Host header 路由请求到对应的反向代理节点：

```
www.scau.edu.cn     → www.scau.edu.cn.s.vpn.scau.edu.cn
other.domain.com    → other.domain.com.s.vpn.scau.edu.cn
```

请求头处理：
- **Host**：替换为 `*.s.vpn.scau.edu.cn` 格式
- **Cookie**：附加 aTrust 会话 cookie（sid, online 等）
- **Connection**：设为 `close`，简化连接管理

### 3.5 会话过期检测

当 aTrust 会话过期时，Web Proxy 返回：
- **HTTP 302** 重定向到 aTrust 门户页面
- **HTTP 502** Bad Gateway（某些情况）

检测方式：
```python
# 解析响应头和响应体
header_end = response.find(b"\r\n\r\n")
resp_head = response[:header_end]
resp_body = response[header_end + 4:]

# 判断是否是 aTrust 返回的页面
is_atrust_page = "aTrust" in body_text or "sf-webproxy" in body_text

# 如果是 302 且是 aTrust 页面，说明会话过期
if status_line.startswith(b"HTTP/1.1 302") and is_atrust_page:
    return 502  # 会话过期
```

## 四、会话保活与自动重连

### 4.1 会话过期原因

aTrust 会话在若干分钟无活动后会过期。过期后：
1. Web Proxy 返回 302 重定向
2. 代理请求得到的是登录页而非目标内容
3. 客户端收到 502 Bad Gateway

### 4.2 保活机制

SessionManager 每 45 秒调用一次 `/passport/v1/user/onlineInfo`：

```python
async def _check_alive(self) -> bool:
    protocol = ATrustProtocol(self._config, session=self._session)
    http = protocol._make_http(self._session)
    data = await http.request("GET", "/passport/v1/user/onlineInfo?...")
    payload = data.json()
    code = payload.get("code", 0)
    return code == 0  # code == 0 表示会话有效
```

### 4.3 自动重连

当检测到会话过期（code == 1000101 或保活失败）时：
1. 调用 `protocol.authenticate()` 重新执行完整的浏览器登录流程
2. 更新 Session 对象中的 cookies、csrf_token 等
3. 通知 WebProxyDialer 更新 cookie
4. 保存会话到 `.session.json`

## 五、TCP 隧道实现（L3 VPN）

### 5.1 何时需要 TCP 隧道

aTrust Web Proxy 只能代理**域名**请求。对于**内网 IP**（如 `222.201.229.3`），需要使用 TCP 隧道。

### 5.2 隧道架构

```
客户端                    TCP 隧道拨号器              aTrust 节点:441
  │                              │                           │
  │─── dial(222.201.229.x) ────▶│                           │
  │                              │─── TLS 连接 ─────────────▶│
  │                              │─── [0x05 init + auth] ──▶│
  │                              │─── [0x05 dest + IP+port]─▶│
  │                              │◀─── [0x05][0x53 OK VIP]──│
  │                              │                           │
  │                              │◀─── L3 心跳连接 ─────────│
  │                              │    (每 25s 发送 0x15)     │
  │                              │                           │
  │◀─── 数据流 (0x05 头 + 原始)──▶│                           │
```

### 5.3 帧格式详解

#### 初始化帧
```
[0x05, 0x01, 0x81, 0x53, 0x03] + [长度(2字节BE)] + [authJSON]
```

authJSON 包含：
```json
{
  "sid": "...",
  "appId": "...",
  "url": "tcp://IP:PORT",
  "deviceId": "...",
  "connectionId": "...",
  "procHash": "...",
  "userName": "...",
  "destAddr": "IP:PORT",
  "env": {...},
  "xRequestSig": "HMAC-SHA256签名"
}
```

#### 目标帧
```
[0x05, 0x01, 0x01, 0x01] + [4字节IP] + [2字节端口BE]
```

#### 响应帧
```
[0x53, 0x00] + [长度(2字节BE)] + [JSON数据]
```
JSON 中 `"OK"` 表示成功，否则是错误信息。

#### SCAU 数据帧（与 zju-connect 不同）
```
[0x05, 0x00, 0x00, 0x01] + [保留6字节] + [原始应用数据]
```
这是 SCAU 特有的帧格式：**固定 10 字节头**，之后直接是原始应用数据（无长度字段）。

### 5.4 L3 心跳连接

aTrust 节点通过 L3 心跳连接判断用户是否在线：

1. 与节点建立 TLS 连接
2. 发送认证帧 `[0x05, 0x01, 0xD0]` + auth JSON
3. 等待认证响应
4. 每 25 秒发送心跳帧 `[0x05, 0x15, 0x00, 0x00]`

如果没有活跃的心跳连接，节点在约 1 分钟后拒绝所有 TCP 隧道请求（错误码 10000004）。

### 5.5 会话刷新

当节点返回错误码 `10000004`（invalid SID）时，需要刷新会话：
1. 调用 `protocol.refresh_session()` 重新认证
2. 获取新的 `sid` cookie
3. 重启 L3 心跳连接

## 七、复现要点总结

### 7.1 认证流程

1. 调用 `/passport/v1/public/authConfig` 获取 csrf_token
2. **使用 Selenium** 打开 CAS 登录页，填写用户名密码（浏览器自动处理 RSA 加密）
3. 访问 `/portal/shortcut.html` 触发会话建立
4. 通过 **CDP** 获取 HttpOnly cookie（CASTGC、sid、online）
5. 调用 `/passport/v1/user/onlineInfo` 确认在线
6. 调用 `/controller/v1/user/clientResource` 获取 IP 资源信息

### 7.2 HTTP 代理

1. 监听本地端口（默认 1081）
2. 接收 HTTP CONNECT 请求
3. 使用本地 CA 签发伪造证书
4. 解密 HTTPS 请求，重写 Host + Cookie
5. 转发到 aTrust Web Proxy
6. 加密响应返回客户端

### 7.3 会话保活

1. 每 45 秒调用 `/passport/v1/user/onlineInfo`
2. 检测到会话过期时，重新执行完整认证流程
3. 更新 Session 和 WebProxyDialer 的 cookies

### 7.4 TCP 隧道（可选）

1. 从 clientResource 获取 IP 资源范围和节点地址
2. 建立 L3 心跳连接（每 25 秒发送 0x15 心跳）
3. 连接时发送 init 帧 + dest 帧
4. 使用 0x05 固定 10 字节头 + 原始数据格式
5. 错误码 10000004 时刷新 sid cookie

## 八、调试技巧

```bash
scau-connect login --no-headless-browser
```


### 8.3 抓包分析

使用浏览器开发者工具或 Fiddler 抓取 `https://vpn.scau.edu.cn` 的请求，分析认证流程。

### 8.4 常见问题

1. **Selenium 启动失败**：确保 Chrome/Edge 已安装
2. **CDP 获取不到 cookie**：确认浏览器已导航到 VPN 域
3. **会话快速过期**：检查保活日志，确保 45s 心跳正常运行
4. **TCP 隧道 10000004**：SID 过期，需要重新认证
